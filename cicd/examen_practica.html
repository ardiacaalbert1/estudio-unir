📝 Tipo test

1. ¿Qué diferencia Entrega Continua (Continuous Delivery) de Despliegue Continuo (Continuous Deployment)?
El producto siempre queda desplegable en ambos. La distinción está en el último salto: Delivery = botón/aprobación manual a prod; Deployment = totalmente automático hasta producción.
2. En la pirámide de pruebas, ¿qué tipo forma la base (más cantidad)?
Base = unitarias (muchas, rápidas, baratas); en medio integración/servicio; arriba E2E/UI (pocas, lentas, caras). Cuanto más arriba, menos pruebas.
3. En un Jenkinsfile declarativo, ¿cuál es la jerarquía correcta?
pipeline { agent ... stages { stage('X') { steps { ... } } } }. El bloque pipeline contiene stages, cada stage contiene steps.
4. En GitLab CI, ¿qué define el archivo .gitlab-ci.yml?
Define stages (orden) y jobs (qué hace cada uno, su script, artifacts, etc.). Los ejecutan los runners de GitLab.
5. Estrategia Blue/Green:
Blue/Green = dos entornos gemelos; el router apunta de azul a verde. Ventaja: rollback inmediato. Coste: duplicar infraestructura.
6. Despliegue Canary:
Canary = exposición gradual a un % creciente de usuarios, vigilando métricas/errores antes de generalizar. Reduce el radio de impacto de un fallo.
7. Sobre los flujos de Git, Gitflow se caracteriza por:
Gitflow define ramas estables (main/develop) y de apoyo (feature/*, release/*, hotfix/*). Es más pesado que trunk-based; por eso muchos equipos prefieren GitLab flow / trunk-based con PRs cortas.
8. En SonarQube, un Quality Gate es:
El Quality Gate marca pasa/falla según umbrales de métricas (p. ej. "0 bugs nuevos", "cobertura ≥ 80% en código nuevo"). Permite romper el pipeline si la calidad baja.
9. ¿Para qué sirve un repositorio de artefactos como Nexus (o Artifactory)?
Guarda los binarios/artefactos resultantes del build y las dependencias, con versionado, para que el pipeline los publique y los entornos los consuman de forma trazable.
10. Los feature flags (feature toggles) permiten:
Desacoplan desplegar de liberar: el código va a producción "apagado" y se enciende por configuración (por usuario, %, entorno). Habilitan canary/A-B y rollback sin redeploy.

✍️ Preguntas cortas / desarrollo

1. Explica las fases típicas de un pipeline de entrega continua.
Fases habituales (de izquierda a derecha):
  • Source / SCM: un commit o merge request dispara el pipeline.
  • Build: compilación y empaquetado del artefacto.
  • Test: pruebas unitarias → integración → análisis estático (SonarQube/quality gate).
  • Release / publicación del artefacto en un repositorio (Nexus).
  • Deploy a entornos (dev → staging → prod) con la estrategia elegida (Blue/Green, Canary, Rolling).
  • Operate / monitor: observabilidad y, si falla, rollback.
Idea clave: cada fase es una puerta de calidad; si una falla, el pipeline se detiene y no se promociona el artefacto.
2. Diferencia entre Blue/Green y Canary. ¿Cuándo usarías cada una?
  • Blue/Green: dos entornos completos; se conmuta el 100% del tráfico de azul a verde. Rollback instantáneo, pero duplica infraestructura. Útil cuando necesitas cambio atómico y vuelta atrás inmediata.
  • Canary: una sola infraestructura; la versión nueva recibe un % pequeño y creciente de tráfico mientras vigilas métricas. Menos coste y riesgo más controlado, pero el despliegue es más lento y exige buena monitorización. Útil para validar en producción con usuarios reales minimizando el impacto de un fallo.
3. ¿Qué es la pirámide de pruebas y por qué la base son las pruebas unitarias?
Modelo que ordena los tipos de prueba por cantidad y coste: base = unitarias (muchas, rápidas, baratas, aíslan una unidad), medio = integración/servicio, cima = E2E/UI (pocas, lentas, frágiles, caras). La base son unitarias porque dan feedback inmediato y localizan el fallo con precisión; invertir la pirámide (muchas E2E) hace los pipelines lentos e inestables.
4. ¿Qué es un Quality Gate en SonarQube y cómo se integra en el pipeline?
Conjunto de condiciones sobre métricas de calidad (bugs, vulnerabilidades, code smells, cobertura y duplicación, normalmente sobre el código nuevo). El análisis de SonarQube se ejecuta como un stage del pipeline; si el Quality Gate da FAILED, el pipeline se marca como fallido y no se promociona el artefacto. Así la calidad se vuelve una condición de despliegue, no una recomendación.

🛠️ Casos prácticos

CASO 1 · estilo Actividad 3
Partes de un Jenkinsfile esqueleto. Añade un stage que ejecute las pruebas y archive el XML de resultados para que Jenkins lo muestre.
pipeline {
  agent any
  stages {
    stage('Build') {
      steps { sh 'mvn -B clean package -DskipTests' }
    }
    stage('Test') {
      steps { sh 'mvn test' }            // genera target/surefire-reports/*.xml
      post {
        always {
          junit 'target/surefire-reports/*.xml'   // publica resultados de test
          archiveArtifacts artifacts: 'target/surefire-reports/*.xml',
                           allowEmptyArchive: true  // archiva el XML
        }
      }
    }
  }
}
Claves que busca el profe: ejecutar las pruebas en su stage, publicar el informe con junit y archivar el XML con archiveArtifacts (en post { always { … } } para que se haga aunque el test falle).
CASO 2
Escribe un .gitlab-ci.yml con tres etapas (build → test → deploy) que solo despliegue desde la rama main.
stages: [build, test, deploy]

build:
  stage: build
  script: ./mvnw -B package
  artifacts:
    paths: [target/*.jar]

test:
  stage: test
  script: ./mvnw test
  artifacts:
    reports:
      junit: target/surefire-reports/TEST-*.xml

deploy:
  stage: deploy
  script: ./deploy.sh target/*.jar
  rules:
    - if: '$CI_COMMIT_BRANCH == "main"'   # solo despliega en main
Claves: stages ordena las fases; cada job declara su stage y script; artifacts pasa el JAR/los informes entre etapas; rules/only limita el deploy a main.

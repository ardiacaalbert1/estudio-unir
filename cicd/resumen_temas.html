Tema 1 · Introducción (ALM y CI/CD)

ALM ⭐ — Application Lifecycle Management: gestión del ciclo de vida completo de la aplicación con tres aspectos: gobierno, desarrollo y operaciones, presentes en cada fase. (pág. 5)Ej.: el gobierno decide qué se construye; desarrollo lo construye; operaciones lo mantiene en producción.
Integración Continua (CI) ⭐ — cada cambio se integra en la rama principal y dispara build + pruebas automáticas. (pág. 17)Ej.: un push a Git lanza un job de Jenkins que compila y ejecuta los tests unitarios.
Entrega Continua vs Despliegue Continuo ⭐ — con Entrega Continua el producto siempre está listo para desplegar pero el paso a producción es manual; con Despliegue Continuo todo cambio que pasa las pruebas llega a producción automáticamente. (págs. 3-4)Ej.: "botón de release" = Delivery; "merge a main = deploy" = Deployment.
Construir/probar/desplegar automáticamente — los cambios de código inician una construcción, ejecutan pruebas y permiten desplegar sin intervención humana (no basta con scripts sueltos o solo jobs). (pág. 5)
Antipatrones — despliegue manual, entornos distintos entre etapas, gestión manual de configuración: lo que la entrega continua elimina. (pág. 5)Ej.: probar el despliegue en producción por primera vez el día de la release.
Modelo de madurez — escala para evaluar cuán automatizado está el proceso (build, test, deploy, medición). (pág. 18)

Tema 2 · Control de versiones

Repositorio — almacén con los ficheros del proyecto y toda su historia de versiones. (pág. 5)
Commit ⭐ — conjunto de cambios registrados en la historia del repositorio. Rama: línea de desarrollo independiente desde un punto de la historia. Merge: combinación de los cambios de dos ramas. (págs. 5-6)Ej. (relacionar T2·P4 del test oficial).
Edición simultánea vs bloqueo — modelo optimista (varios editan y luego se fusionan) frente a bloquear el fichero para un solo editor; Git usa el primero. (págs. 6-9)
Qué versionar ⭐ — código, scripts, configuración y definición del pipeline; los binarios generados NO (van al repositorio de artefactos). (Tema 2)

Tema 3 · Flujos de trabajo con Git

Flujo centralizado ⭐ — una sola rama principal (trunk/master); todos hacen commit ahí. (pág. 5)
Feature branching ⭐ — una rama por funcionalidad en el mismo repo; se fusiona al terminar. Puede implementarse en cualquier VCS (no exige pull requests). (pág. 10)
Gitflow ⭐ — ramas permanentes main y develop + feature/release/hotfix. Las features se fusionan a develop; cada commit en master se etiqueta con la versión. (pág. 12)
GitLab flow — feature branches + ramas de entorno (production, pre-production): el despliegue va ligado a ramas. (pág. 18)
Forking workflow — cada desarrollador con su fork y contribución vía pull requests entre forks (típico de open source). (págs. 6, 16)

Tema 4 · Fases de la entrega continua

Pipeline de despliegue ⭐ — cadena automatizada: source → build → test → release → deploy → operate; cada fase es una puerta de calidad que detiene el avance si falla. (pág. 4)
Fase de commit ⭐ — la primera: compila y ejecuta pruebas unitarias; da feedback en minutos. (pág. 6)
Artefacto único — se construye una sola vez y se promociona el mismo binario por todos los entornos. (pág. 6)Ej.: el JAR que pasó las pruebas es el que llega a producción, no una recompilación.
Pruebas de aceptación — validan requisitos de negocio en un entorno tipo producción, tras la fase de commit. (pág. 7)

Tema 5 · Pruebas

Pirámide de pruebas ⭐ — base: unitarias (muchas, rápidas, baratas) · medio: integración/servicio · cima: E2E/UI (pocas, lentas, caras). (pág. 11)
Tipos de prueba ⭐ — unitarias (una unidad aislada) · de sistema (E2E con la app arrancada) · de GUI (vía navegador) · de rendimiento (no funcionales: tiempos bajo carga). (págs. 6-8)Ej. (relacionar T5·P5 del test oficial).
Informes JUnit ⭐ — ficheros XML con el resultado de la ejecución de pruebas; los publican las herramientas de CI. No son exclusivos de Java ni de pruebas unitarias. (pág. 25)
Automatización — las pruebas se ejecutan en el pipeline en cada cambio; las manuales quedan para exploratorias. (Tema 5)

Tema 6 · Evaluación y calidad del código

Métricas de entrega — lead time, frecuencia de despliegue, tasa de fallos, tiempo de recuperación: miden la salud del proceso. (pág. 6)
Análisis estático — examinar el código sin ejecutarlo buscando bugs, vulnerabilidades y code smells. (pág. 5)
SonarQube ⭐ — plataforma de análisis estático: calidad desde varios puntos de vista + evolución temporal + problemas potenciales ("todas las anteriores" en el test). (pág. 4)
Quality Gate ⭐ — condiciones (bugs, cobertura, duplicación… sobre código nuevo) que deciden pasa/falla; integrado como stage, rompe el pipeline si falla. (Tema 6)
Revisiones de código — inspección por pares (pull requests) como práctica de calidad complementaria al análisis automático. (pág. 4)

Tema 7 · Jenkins

Arquitectura master + agentes ⭐ — el master orquesta (UI, cola, planificación) y los agentes/nodos ejecutan los trabajos. (pág. 4)
Jenkinsfile ⭐ — pipeline como código (Groovy) versionado en el repositorio. Es donde "debe definirse" el pipeline. (pág. 4)
Declarativa vs scripted — declarativa: estructura fija pipeline → stages → stage → steps; scripted: Groovy libre. (pág. 14)
Triggers — qué dispara el pipeline: webhook del SCM, encuesta (polling), programado (cron), manual. (pág. 7)
post { always { … } } ⭐ — acciones tras el stage pase lo que pase: publicar junit y archiveArtifacts (clave del ejercicio práctico estilo Actividad 3). (Tema 7)

Tema 8 · GitLab

Plataforma integrada — repositorio + CI/CD + registro en un solo producto. (Tema 8)
.gitlab-ci.yml ⭐ — YAML en la raíz del repo que define stages (orden) y jobs (script, artifacts, rules). (pág. 8)
Runners ⭐ — procesos que ejecutan los jobs; executors: shell, Docker, Kubernetes. (pág. 8)
Artifacts — ficheros que un job guarda y pasa a los siguientes (JAR, informes JUnit). (pág. 21)
Disparadores ⭐ — el disparo general del pipeline es el push a una rama (commits/tags); también merge requests. (Tema 8)

Tema 9 · Otras herramientas

Bitbucket ⭐ — repositorios Git de Atlassian (históricamente también Mercurial), integrado con Jira/Trello; CI con Bitbucket Pipelines. (pág. 5)
CircleCI — CI/CD en la nube; destaca por las cachés para acelerar builds; config en .circleci/config.yml. (pág. 7)
Nexus ⭐ — repositorio de artefactos: almacena/versiona binarios (JAR, imágenes, paquetes) y hace de proxy de dependencias. (pág. 23)

Tema 10 · Modelos de despliegue

Blue/Green ⭐ — dos entornos idénticos; se conmuta TODO el tráfico de golpe; rollback inmediato; coste: infraestructura doble. (pág. 10)
Canary releasing ⭐ — la nueva versión recibe una porción pequeña y creciente del tráfico real; un fallo solo afecta a esa porción; exige monitorización. (pág. 12)
Rolling (incremental) ⭐ — se sustituyen los servidores poco a poco; ambas versiones conviven durante el proceso. (págs. 12-13)
Feature flags ⭐ — activar/desactivar funcionalidades por configuración sin desplegar; desacoplan despliegue de exposición; habilitan A/B y rollback sin redeploy. (pág. 4)

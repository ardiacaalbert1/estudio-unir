CI/CD — Chuleta de examen

Examen: 8 test (4 pts) + 1 desarrollo teórico-práctico + 1 ejercicio práctico. Los test son similares a los de la plataforma. El práctico gira en torno a Jenkins (estilo Actividad 3).

📌 Definiciones que caen seguro

  • ALM = gobierno + desarrollo + operaciones.
  • CI: integrar y probar automáticamente cada cambio.
  • Entrega Continua: siempre desplegable; paso a producción manual.
  • Despliegue Continuo: a producción automático si pasan las pruebas.
  • Construir/probar/desplegar automáticamente = los cambios de código inician build + pruebas + despliegue sin intervención humana.
  • Informes JUnit = XML con resultados de ejecución de pruebas (no solo Java/unitarias).
  • Quality Gate (Sonar): condiciones que el código debe cumplir o el pipeline falla. Sonar aporta: calidad multi-vista + evolución + problemas potenciales (todas).

🌿 Git y flujos (T2-T3)

  • Commit: cambios registrados en la historia · Rama: línea independiente · Repo: ficheros + historia · Merge: combinar dos ramas.
  • Centralizado: solo trunk/master (1 rama).
  • Feature branching: rama por funcionalidad, mismo repo; vale en cualquier VCS (no exige PRs).
  • GitLab flow: features + ramas de entorno (production/pre-production).
  • Forking: fork por dev + PRs entre forks.
  • Gitflow: main + develop + feature/release/hotfix. Features → develop. Cada commit en master lleva etiqueta de versión.
  • Versionar código/scripts/config; no los binarios generados (→ Nexus).

🧪 Pruebas y pipeline (T4-T5)

  • Pipeline: source → build → test → release → deploy → operate. Cada fase = puerta de calidad.
  • Fase de commit: build + pruebas unitarias.
  • Pirámide: base unitarias (muchas/rápidas) → integración → cima E2E/UI (pocas/lentas).
  • Tipos: unitarias (aisladas) · sistema (E2E, app arrancada) · GUI (navegador) · rendimiento (no funcional).

⚙️ Herramientas (T7-T9)

  • Jenkins: master + agentes. Pipeline como código en Jenkinsfile (repo). Jerarquía: pipeline → stages → stage → steps.
  • GitLab CI: .gitlab-ci.yml define stages y jobs; los ejecutan runners (shell/Docker/K8s). Disparo general: push a rama.
  • Bitbucket: Git de Atlassian (+ Jira/Trello). Históricamente también Mercurial.
  • CircleCI: CI/CD cloud, cachés.
  • Nexus: repositorio de artefactos + proxy de dependencias.

🛠️ Código que pide el práctico

Jenkinsfile — test + archivar XML (estilo Act. 3): 
pipeline {
  agent any
  stages {
    stage('Build') { steps { sh 'mvn -B clean package -DskipTests' } }
    stage('Test') {
      steps { sh 'mvn test' }
      post { always {
        junit 'target/surefire-reports/*.xml'
        archiveArtifacts artifacts: 'target/surefire-reports/*.xml',
                         allowEmptyArchive: true
      } }
    }
  }
}
.gitlab-ci.yml — build/test/deploy solo en main: 
stages: [build, test, deploy]
build: { stage: build, script: ./mvnw -B package,
         artifacts: { paths: [target/*.jar] } }
test:
  stage: test
  script: ./mvnw test
  artifacts: { reports: { junit: target/surefire-reports/TEST-*.xml } }
deploy:
  stage: deploy
  script: ./deploy.sh target/*.jar
  rules: [ if: '$CI_COMMIT_BRANCH == "main"' ]

📝 Tipo test

1. En un Dockerfile, ¿qué diferencia hay entre CMD y ENTRYPOINT?
ENTRYPOINT = el programa que se ejecuta siempre; CMD = sus argumentos por defecto. docker run img args sustituye al CMD, no al ENTRYPOINT (salvo --entrypoint).
2. ¿Por qué importa el orden de las instrucciones en un Dockerfile?
Las imágenes son capas. Si una capa cambia, se invalida la caché de todas las siguientes. Por eso se copia primero package.json/pom.xml e instalan dependencias, y el código fuente (que cambia mucho) al final.
3. Diferencia entre un volumen y un bind mount:
Volumen = almacenamiento gestionado por Docker, portable y recomendado para datos persistentes. Bind mount = ruta literal del host montada dentro del contenedor (útil en desarrollo, frágil entre máquinas).
4. ¿Para qué sirve Docker Compose?
Compose describe en YAML varios servicios y sus redes/volúmenes para levantarlos juntos con docker compose up. Para orquestación multinodo en prod se usa Swarm o Kubernetes.
5. En Kubernetes, un Pod es:
El Pod es la unidad mínima que K8s programa. Sus contenedores comparten el mismo network namespace (una IP, localhost entre ellos) y pueden compartir volúmenes.
6. ¿Para qué se usan las labels y los selectors?
Las labels (app: web) etiquetan objetos; los selectors los seleccionan. Así un Service enruta tráfico a los Pods cuyo label coincide. El profe las marca como "lo más importante" en Pods.
7. Diferencia entre liveness probe y readiness probe:
Liveness falla → K8s reinicia el contenedor. Readiness falla → K8s lo saca de los endpoints del Service (no le manda tráfico) sin reiniciarlo. El profe las marca como "lo más importante".
8. ¿Qué tipo de Service expone la aplicación fuera del clúster mediante un balanceador externo?
ClusterIP (solo interno, por defecto) → NodePort (abre un puerto en cada nodo) → LoadBalancer (provisiona un balanceador externo del proveedor). Para HTTP suele añadirse un Ingress por delante.
9. La relación correcta en un despliegue de Kubernetes es:
El Deployment declara el estado deseado y gestiona un ReplicaSet, que asegura el número de réplicas de Pods. Al cambiar la imagen hace rolling update y permite rollout undo.
10. Un CronJob en Kubernetes sirve para:
Un Job ejecuta una tarea hasta completarse; un CronJob lanza Jobs según un horario cron ("0 2 * * *" = cada día a las 2:00). El profe avisó: posible pregunta corta sobre CronJobs.

✍️ Preguntas cortas / desarrollo

1. Explica la diferencia entre CMD y ENTRYPOINT con un ejemplo.
ENTRYPOINT fija el ejecutable; CMD aporta argumentos por defecto sobrescribibles. 
ENTRYPOINT ["ping"]
CMD ["localhost"]
docker run img → ping localhost. docker run img google.com → ping google.com (se sustituye el CMD, no el ENTRYPOINT). Patrón habitual: ENTRYPOINT = comando fijo, CMD = parámetros por defecto.
2. Liveness vs Readiness probe: para qué sirve cada una.
  • Liveness: comprueba si el contenedor sigue funcionando. Si falla repetidamente, K8s lo reinicia. Detecta cuelgues/deadlocks.
  • Readiness: comprueba si el Pod está listo para atender peticiones. Si falla, K8s lo quita de los endpoints del Service (deja de enviarle tráfico) pero no lo reinicia. Útil al arrancar o ante dependencias temporalmente caídas.
(Existe además la startup probe para arranques lentos.)
3. la marcó el profe Explica el uso de los CronJobs en Kubernetes. Pon un ejemplo.
Un CronJob ejecuta Jobs de forma periódica según una expresión cron. Sirve para tareas recurrentes: copias de seguridad, limpieza, informes, sincronizaciones. 
apiVersion: batch/v1
kind: CronJob
metadata: { name: backup }
spec:
  schedule: "0 2 * * *"        # cada día a las 02:00
  jobTemplate:
    spec:
      template:
        spec:
          restartPolicy: OnFailure
          containers:
          - name: backup
            image: alpine
            command: ["sh","-c","echo haciendo backup"]
Campos clave: schedule (cron), jobTemplate (el Job a lanzar) y restartPolicy (OnFailure/Never).
4. Tipos de Service en Kubernetes y cuándo usar cada uno.
  • ClusterIP (por defecto): IP interna; comunicación entre Pods dentro del clúster.
  • NodePort: abre el mismo puerto en todos los nodos; acceso externo básico (pruebas).
  • LoadBalancer: balanceador externo del proveedor cloud; exposición a internet.
  • ExternalName: alias DNS a un servicio externo.
Para HTTP/HTTPS con rutas y dominios se suele poner un Ingress delante de Services ClusterIP.

🛠️ Casos prácticos

CASO 1 · Dockerfile
Escribe un Dockerfile multi-stage para una app Node: compila en una etapa y deja una imagen final ligera que ejecute node dist/server.js.
# Etapa de build
FROM node:20 AS build
WORKDIR /app
COPY package*.json ./
RUN npm ci                 # deps primero → mejor caché
COPY . .
RUN npm run build          # genera /app/dist

# Etapa final ligera
FROM node:20-alpine
WORKDIR /app
COPY package*.json ./
RUN npm ci --omit=dev
COPY --from=build /app/dist ./dist
EXPOSE 3000
CMD ["node", "dist/server.js"]
Claves: multi-stage para no arrastrar herramientas de build, COPY package*.json antes que el código (caché), imagen final -alpine y solo dependencias de producción.
CASO 2 · estilo Práctica 3
Despliega en Kubernetes una app web con 3 réplicas y exponla dentro del clúster. Escribe el Deployment y el Service.
apiVersion: apps/v1
kind: Deployment
metadata: { name: web }
spec:
  replicas: 3
  selector:
    matchLabels: { app: web }     # casa con el label del Pod
  template:
    metadata:
      labels: { app: web }
    spec:
      containers:
      - name: web
        image: nginx:1.27
        ports: [{ containerPort: 80 }]
        readinessProbe:
          httpGet: { path: /, port: 80 }
---
apiVersion: v1
kind: Service
metadata: { name: web-svc }
spec:
  selector: { app: web }          # enruta a los Pods con app=web
  ports:
  - port: 80
    targetPort: 80
  type: ClusterIP
Claves que valora el profe: el selector del Service y el matchLabels del Deployment deben coincidir con las labels del Pod; replicas: 3; una readiness probe para no enrutar a Pods no listos.
